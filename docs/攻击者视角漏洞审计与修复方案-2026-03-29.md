# Weblog 攻击者视角漏洞审计与修复方案（2026-03-29）

## 1. 范围与方法

- 范围：`weblog-backend`（认证、限流、验证码、访问控制、上传、搜索）+ `weblog-user`（内容渲染与前端防护）。
- 方法：代码审计 + 本地黑盒回归（`curl/python requests`）模拟攻击者请求。
- 结论分级：
  - `已证实`：有代码证据，且已通过本地请求复现。
  - `高风险待验证`：有明确代码风险信号，但未完成完整可利用链复现。

## 3. 详细问题与修复方案

## V-01 未登录阅读限制可绕过（已证实，高）

**事实（证据）**

- 指纹由 `User-Agent + IP` 直接生成：
  - `weblog-backend/weblog-api/src/main/java/com/blog/api/portal/AccessControlController.java:88`
  - `weblog-backend/weblog-module-interaction/src/main/java/com/blog/interaction/service/AccessControlService.java:117`
- 代码对 UA 可控值无抗伪造机制（无设备绑定 Cookie、无签名、无挑战）。

**本地复现（已执行）**

- 同一 IP 下，`User-Agent: attack-ua-a` 连续记录 3 篇后 `allowed=false`；
- 切换为 `User-Agent: attack-ua-b` 立刻回到 `allowed=true`、`readCount=0`。

**攻击影响**

- 任意未登录用户可绕过“每日限读”策略。
- 访问控制失效，业务限流价值下降。

**修复方案（P0）**

1. 指纹从 `UA+IP` 改为 `server-side deviceId cookie + IP段 + UA摘要`：
   - 首次访问下发 `HttpOnly + SameSite=Lax/Strict + Secure` 的随机 `device_id`。
   - 服务端对 `device_id` 做 HMAC 签名并校验，防篡改。
2. `recordRead` 与 `checkAccess` 都基于签名 `device_id`，UA 仅作为辅助熵，不再作为主键。
3. 访问控制键增加行为防刷阈值（每设备每日最大唯一文章计数硬上限）。

**验收标准**

- 仅修改 UA 不再影响同设备阅读计数。
- 删除 cookie 后重新生成新设备身份（计数可重置，符合预期）。

## V-02 限流/验证码白名单绕过风险（已证实，高）

**事实（证据）**

- 限流切面对白名单 IP 直接跳过：
  - `weblog-backend/weblog-infra-security/src/main/java/com/blog/infra/security/ratelimit/RateLimitAspect.java:48`
  - `weblog-backend/weblog-infra-security/src/main/java/com/blog/infra/security/ratelimit/RateLimitAspect.java:71`
- 验证码黑名单与失败计数对白名单 IP 不生效：
  - `weblog-backend/weblog-infra-captcha/src/main/java/com/blog/infra/captcha/service/CaptchaServiceImpl.java:39`
  - `weblog-backend/weblog-infra-captcha/src/main/java/com/blog/infra/captcha/service/CaptchaServiceImpl.java:232`
  - `weblog-backend/weblog-infra-captcha/src/main/java/com/blog/infra/captcha/service/CaptchaServiceImpl.java:239`
- 本地实测：搜索接口 45 次连续请求均成功（无 429）。

**风险说明**

- 若生产流量在应用层呈现为 `127.0.0.1/::1`（典型反代误配），限流与验证码防刷会整体失效。

**修复方案（P0）**

1. 删除代码硬编码白名单逻辑，改为配置开关：
   - `blog.security.dev-bypass-enabled`，且仅 `dev/test` profile 可开启。
2. 统一基于 `IpUtil` + 可信代理链提取真实客户端 IP；生产环境强制校验 `TRUSTED_PROXY_IPS`。
3. 启动时增加安全自检：
   - 若 `prod` 且 `dev-bypass-enabled=true`，启动失败。
4. 对认证关键接口（登录/验证码/发送验证码/忘记密码）增加“用户维度 + IP 维度”双桶限流。

**验收标准**

- `prod` 下白名单绕过不可生效。
- 认证接口超频触发统一 `RATE_LIMIT` 返回码。

## V-03 check-email 存在 DNS 查询滥用面（已证实，中）

**事实（证据）**

- 公开接口直接用用户输入邮箱域名发起 DNS MX 查询：
  - `weblog-backend/weblog-api/src/main/java/com/blog/api/portal/AuthController.java:186`
  - `weblog-backend/weblog-api/src/main/java/com/blog/api/portal/AuthController.java:196`
- 此接口未要求验证码，仅有基础限流。

**攻击影响**

- 可被批量用于域名探测、DNS 资源消耗、内部 DNS 信息侧探（取决于部署 DNS 策略）。

**修复方案（P1）**

1. 对 `check-email` 增加滑块验证码校验（与 `send-code/login/register` 同级）。
2. 限流升级为：`IP + 邮箱域名` 复合键，窗口更严格（例如 3/分钟）。
3. 增加域名规则：
   - 仅允许标准公网域名；拒绝内网后缀、单标签域名、超长域名。
4. 增加 DNS 查询缓存（短 TTL）与熔断，避免被放大打满。

**验收标准**

- 无验证码时调用直接拒绝。
- 同 IP 高频域名探测迅速触发限流。

## V-04 阅读记录接口可被用于 Redis 放大写入（已证实，中）

**事实（证据）**

- `recordRead` 对 `postId` 不做存在性校验，直接写集合：
  - `weblog-backend/weblog-api/src/main/java/com/blog/api/portal/AccessControlController.java:63`
  - `weblog-backend/weblog-module-interaction/src/main/java/com/blog/interaction/service/AccessControlService.java:67`

**攻击影响**

- 攻击者可用大量伪造 `postId` 写入每日集合，造成 Redis 键值膨胀。

**修复方案（P1）**

1. `recordRead` 前校验文章是否存在且可访问（发布且未禁用）。
2. 单设备每日唯一文章数设置硬上限（例如 200），超限后拒绝写入。
3. key 过期时间改为“到次日零点”，并加后台清理兜底任务。

**验收标准**

- 不存在 `postId` 不写 Redis。
- 超过上限返回明确错误且不再增长集合。

## V-05 Markdown 清洗链路存在存储型 XSS 风险信号（高风险待验证，高）

**事实（证据）**

- 后端 Markdown 清洗采用正则替换，不是结构化 HTML Sanitizer：
  - `weblog-backend/weblog-infra-security/src/main/java/com/blog/infra/security/util/XssUtil.java:52`
- 文章内容在前端通过 `MdPreview` 渲染，当前页面未见统一 DOMPurify 二次净化：
  - `weblog-user/pages/post/[slug].vue:71`
- 文章保存时走 `cleanMarkdown`：
  - `weblog-backend/weblog-module-content/src/main/java/com/blog/content/service/PostService.java:115`

**推测/假设（依据）**

- 若 Markdown 渲染器允许部分 HTML/协议而后端正则未覆盖，存在存储型 XSS 可能。

**修复方案（P0）**

1. 后端将 `cleanMarkdown` 从正则替换升级为成熟策略：
   - 使用 `OWASP Java HTML Sanitizer`（严格白名单）或统一 `Jsoup Safelist` 结构化净化。
2. 前端文章渲染链路增加二次净化（DOMPurify）或启用渲染器安全模式。
3. 建立恶意载荷回归集（`svg/onload`、`math`、`xlink:href`、编码绕过、协议绕过）。

**验收标准**

- 恶意 Markdown/HTML 样本渲染后不执行脚本。
- 页面功能与合法富文本展示不回归。

## 5. 回归测试清单（必须自动化）

1. 同 IP 切换 UA 不得重置阅读计数。
2. 登录/验证码/发送验证码在高并发下必须触发限流。
3. `check-email` 无验证码请求必须被拒绝。

5. 恶意 Markdown 样本渲染不得执行脚本。
6. `prod` 环境若开启开发绕过开关必须启动失败。
